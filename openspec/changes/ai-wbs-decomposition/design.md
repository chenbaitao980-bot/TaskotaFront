# AI WBS 分解 — 设计方案

## 背景
当前 AI 分解只能拆出固定模板（"调研/方案/实现/测试/文档"），对于复杂任务描述返回不可用结果。需要参考业界 WBS 最佳实践重构。

## 调研结论

| 来源 | 关键方案 | 可借鉴点 |
|------|----------|----------|
| CMU 2025 (LLM Decomposition) | 将分解组件集成到 SWE-agent + 24% SWE-Bench提升 | **Classify → Decompose → Execute** 三步流水线 |
| AlphaBOLD 工程咨询案例 | 结构化 prompt 替代 RAG，5 分钟完成之前数天的 WBS | **分阶段 prompt chain**，按角色引导输出 |
| Least-to-Most Prompting (Google Brain) | 两阶段：分解成子问题 → 从简到难依次求解 | **递归分解**，每级控制 2-5 个子任务 |
| Runtime-Structured Decomposition (arXiv 2026) | 静态分解可能增加重试成本，**运行时动态分解**更优 | 失败时只重试失败子任务 |
| Prompt Chain 设计 (GitHub) | 5 种分解模式：线性、分支、递归、模块、迭代 | **DAG 依赖建模** |

## 架构设计

```
┌──────────────────────────────────────────────┐
│            AI WBS 分解服务                      │
│                                                │
│  描述文本 → [理解层] → [分解层] → [组装层] → 子树   │
│                                                │
│  理解层：提取意图、范围、关键里程碑                   │
│  分解层：递归拆解，每级 2-5 项，最多 3 级           │
│  组装层：将结构化的 JSON 转为 SubtaskNode 树       │
└──────────────────────────────────────────────┘
```

### Prompt 策略（三阶段）

**Phase 1 — 理解**：识别任务类型、复杂度、关键约束
```
角色：项目管理专家
任务：分析以下任务描述，提取：任务类型、关键里程碑、预计复杂度
```

**Phase 2 — 分解**：按 WBS 原则递归拆解
```
角色：WBS 分解专家  
规则：
1. 每个节点拆 2-5 个子项
2. 最多 3 层深度
3. 子任务用"动词+名词"格式
4. 严格遵守描述中的顺序和层级关系
5. "先做A，做完C和D再去做B" → A(一级)，B(一级，C和D为B的子任务)
输出格式：{"subtasks":[{"title":"动词+名词","children":[]}]}
```

**Phase 3 — 验证**：检查完整性和一致性
```
角色：QA 审核员
检查：所有关键里程碑是否都被覆盖？是否有冗余或遗漏？
```

### 关键技术决策

| 决策 | 选择 | 原因 |
|------|------|------|
| API 模型 | DeepSeek Chat | 已有 API Key，兼容 OpenAI 格式 |
| Temperature | 0.1 | 低温度保证一致性 |
| 最大 Token | 4096 | 复杂任务需要更大输出空间 |
| 重试策略 | 失败时退化为关键词匹配 fallback | 避免显示空结果 |
| 输出格式 | JSON + 嵌套 children | 天然支持树形结构 |

### 预期效果

| 输入 | 当前输出 | 预期输出 |
|------|----------|----------|
| "先做A，做完C和D再去做B" | 方案设计/核心实现... | A, B(C, D) |
| "今年拿下日语N1，先去报名然后复习N1资料" | 测试12 | 报名N1考试, 复习N1资料 |
| 复杂软件开发任务 | 固定模板 | 按描述动态拆解 |
