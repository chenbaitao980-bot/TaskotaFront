# 方案：任务管理系统重构

## 一、背景与问题分析

### 1.1 当前问题

当前系统的任务管理基于 `TaskBreakdown` 实体，存在以下问题：

1. **层级体系过度设计**：`yearly/quarterly/monthly/weekly/daily/hourly` 六级体系源自目标管理（OKR/KPI 分解），不适合通用任务管理场景。用户日常需要的是「项目 - 任务 - 子任务」三层结构，而非时间粒度分解。

2. **数据层混乱**：
   - `LocalStorageService`（SharedPreferences JSON）与 `SupabaseService` 双轨并行
   - 页面直接调用 `LocalStorageService`，BLoC 却调用 `SupabaseService`，没有统一抽象
   - SharedPreferences 全量读写 JSON，数据量增大后性能急剧下降
   - 没有类型安全的查询能力，全部是内存遍历过滤

3. **UI 与模型耦合过深**：任务列表页、详情页、创建页强耦合于 `TaskBreakdown` 模型，修改模型需要改动大量页面代码。

### 1.2 参考对象：滴答清单（TickTick）

滴答清单采用三层数据模型：
- **项目（Project）**：任务容器，支持颜色、排序、归档
- **任务（Task）**：具体工作项，含优先级、日期、检查项、提醒、重复
- **检查项（ChecklistItem）**：子任务勾选

交互上采用「项目分组 → 任务列表 → 任务详情」的渐进式展开，广泛采用滑动操作、快速编辑、批量处理。

### 1.3 影响范围（基于 GitNexus 分析）

| 维度 | 数据 |
|------|------|
| 风险等级 | CRITICAL |
| 影响符号数 | 29 |
| 影响执行流 | 5 |
| 影响模块 | Entities、Calendar、Services、Ai_chat |

> 本次独立模块不涉及 Calendar 和 Ai_chat 的改造，风险集中在新增文件和导航改造。

## 二、解决方案概览

### 2.1 核心策略

**新建独立任务管理模块**，与现有 TaskBreakdown 系统并行运行。现有日历和 AI 聊天仍使用旧系统，新模块专注做好任务管理本身。

| 层面 | 现状 | 目标 |
|------|------|------|
| 数据模型 | TaskBreakdown（六级层级） | Project / Task / ChecklistItem（滴答简化版） |
| 数据库 | SharedPreferences JSON | drift（sqflite 本地数据库） |
| 数据访问 | 双轨并行无抽象 | Repository 统一接口 |
| 状态管理 | 页面直接调 Service | BLoC 统一管理 |
| 导航 | 无独立任务入口 | 底部导航第二位「任务」Tab |
| 页面 | 现有 TaskList/Detail/Create | 全新 UI，参考滴答交互 |

旧系统（TaskBreakdown、LocalStorageService、旧任务页面）**全部保留不动**，待日历和 AI 聊天未来重构后再统一清理。

### 2.2 简要数据模型

```
Project（项目）
  └── Task（任务，多对一）
        └── ChecklistItem（检查项，一对多）
```

各模型字段采用简化版滴答风格，一期去掉时区、重复规则、提醒等复杂能力。

### 2.3 导航结构

底部导航从 4 个 Tab 扩展为 5 个：

**首页 → 任务 → 日历 → AI助手 → 我的**

新增的「任务」Tab 作为独立模块入口，承载项目分组和任务列表。

## 三、范围

### 3.1 本次做（In Scope）

1. **数据层（全新）**
   - 引入 drift 作为本地数据库
   - 创建 `projects`、`tasks`、`checklist_items` 三张表
   - 实现 Repository 统一数据访问层

2. **新任务管理 UI（全新）**
   - 「任务」Tab 入口（底部导航第二位）
   - 项目侧边栏（左侧抽屉式，类似滴答的项目列表）
   - 任务列表视图（按项目筛选，支持状态/优先级/日期排序分组）
   - 任务详情页（任务信息展示、检查项勾选管理）
   - 创建/编辑任务表单（项目选择、标题、描述、优先级、日期、检查项）
   - 快速操作：左滑完成/删除、点击编辑

3. **底部导航改造**
   - HomePage 底部导航从 4 项扩展为 5 项
   - 新增「任务」Tab，导航至新 TasksPage

### 3.2 本次不做（Out of Scope）

- ❌ 日历集成（后期重构）
- ❌ AI 聊天集成（后期重构）
- ❌ 删除旧 TaskBreakdown / 旧任务代码（仍需支持日历和 AI 聊天）
- ❌ Supabase 云端同步
- ❌ 提醒/通知系统
- ❌ 标签系统（Tags）
- ❌ 附件上传 / 评论系统
- ❌ 重复周期任务（Recurring Tasks）
- ❌ 时区支持 / 富文本备注
- ❌ 看板视图 / 分享与协作

## 四、关键取舍

| 决策 | 选择 | 理由 |
|------|------|------|
| 新旧关系 | 独立并行，不删旧代码 | 日历和 AI 聊天仍依赖旧系统，避免破坏现有功能 |
| 数据库 | drift (sqflite) | 类型安全、支持 SQL 查询、成熟稳定、Flutter 生态首选 |
| 状态管理 | BLoC（延续现有模式） | 项目已使用 BLoC，保持一致性 |
| 数据模型 | 扁平 Task + Project 分组 | 简化层级，适配通用任务管理 |
| 项目选择 | 左侧滑出面板（Drawer） | 滴答风格，节省主屏幕空间 |
| 检查项 | 内联在任务详情页展开/折叠 | 避免页面跳转层级过深 |
| 云端同步 | 第一期纯本地 | 降低第一期复杂度，后续通过 Repository 抽象接入 |

## 五、验收标准

1. ✅ 用户可在底部导航点击「任务」进入任务管理模块
2. ✅ 用户可创建/编辑/删除项目（Project）
3. ✅ 用户可在项目中创建/编辑/删除任务（Task）
4. ✅ 用户可在任务中添加/编辑/删除检查项（ChecklistItem）
5. ✅ 用户可勾选任务和检查项标记完成
6. ✅ 任务列表支持按项目、优先级、日期筛选排序
7. ✅ drift 数据库读写正常，数据持久化可靠
8. ✅ 旧系统功能不受影响（日历、AI 聊天、旧任务页面正常运行）

## 六、风险与缓解

| 风险                                     | 等级  | 缓解措施                                     |
| -------------------------------------- | --- | ---------------------------------------- |
| 新建 drift 数据库与旧 SharedPreferences 数据不一致 | 中   | 新模块使用独立数据库，不依赖旧数据，完全隔离                   |
| 底部导航 Tab 增加可能影响首页状态管理                  | 中   | HomePage 已有 IndexedStack 模式，扩展 4→5 项风险可控 |
| 用户有两套任务系统造成困惑                          | 低   | UI 上明确区分：底部导航「任务」是新模块，旧入口逐步淡化；后期统一后清理旧系统 |
