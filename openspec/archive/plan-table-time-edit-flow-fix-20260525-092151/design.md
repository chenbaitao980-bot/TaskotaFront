# 设计：plan-table-time-edit-flow-fix

## 当前问题
当前计划表数据结构只有 `stage/task/note` 三段，实际承载了日期、主题、训练内容等混合信息。由于缺少独立的开始/结束时间字段，界面只能显示日期或阶段，无法精确到分钟。

流程图展示时存在节点数量限制，导致计划内容较多时只显示一部分。截图中的一键分配预览还显示第一个任务标题为“主题”，说明表头行或主题字段被错误当成了任务标题。

计划表目前主要是展示组件，用户无法直接在计划表里修改时间、主题和训练内容，只能在分配弹窗里部分修改。

## 方案
1. 先定位当前计划解析、计划表渲染、流程图渲染和任务草稿生成符号，按 GitNexus 要求做影响分析。
2. 扩展计划行的内存表示，至少能表达：开始时间、结束时间、主题、训练内容。若不改类名，也需要保证字段语义清晰。
3. 解析 Markdown 表格时跳过表头行，避免“主题”进入任务标题。
4. 计划表渲染改为可编辑控件，用户可以直接修改开始/结束时间、主题和训练内容。
5. 流程图渲染移除固定 7 条限制，使用横向滚动或换行布局展示全部节点。
6. 一键分配任务草稿使用用户编辑后的计划行：标题来自主题或动作标题，说明来自训练内容，开始/结束时间来自计划表时间。

## 风险控制
- 不修改本地存储接口和任务实体结构。
- 不修改日历视图的渲染逻辑。
- 如果现有 `_PlanRow` 影响范围较大，优先在 AI 聊天页内部增加兼容字段和转换方法，减少跨文件影响。

## 验证
- 使用包含 11 个计划节点的样例，确认流程图全部展示。
- 使用包含“日期、主题、训练内容、开始时间、结束时间”的样例，确认表格展示到分钟。
- 打开一键分配弹窗，确认任务标题不出现“主题”表头。
- 在计划表直接编辑时间、主题和训练内容，再打开一键分配，确认弹窗使用编辑后的值。
