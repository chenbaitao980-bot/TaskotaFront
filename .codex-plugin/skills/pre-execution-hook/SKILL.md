# Reasonix 全局记忆预执行钩子

## 概述

本钩子整合 `.reasonix/memory/global/` 全部 22 条记忆规则，分为 6 个阶段。每次执行代码/文件/配置/构建/测试任务前，必须完整走完所有阶段。

---

## 一、回复格式 (Reply Format)

### 1.1 记忆执行状态头
```
[记忆执行情况] 已读 22 已执行 22 ✅
```

### 1.2 前缀 [老板]
```
[老板]

(正文)
```

### 1.3 外部触发 "格式" 命令
当用户说「格式」时，立即停止当前推理 → 检查回复是否满足 1.1+1.2 → 不满足则删除草稿重写，满足则在正文前加"格式自检通过"。

---

## 二、改代码前门禁 (Pre-Edit Gate)

### 2.1 工具可用性检查
每次 edit_file / multi_edit / write_file 前，第一段必须是：
```
[图谱工具] 可用工具检查
  codegraph MCP:
    - codegraph_impact:   ✅/❌(本轮 <符号名> 实际调用过/从未调用过)
    - codegraph_explore:  ✅/❌(本轮 <符号名> 实际调用过/从未调用过)
    - codegraph_callers:  ✅/❌(本轮 <符号名> 实际调用过/从未调用过)
  graphify 索引:          ✅/❌
```
任一 ❌ → 禁止修改代码。禁止用 read_file / search_content 替代图谱语义分析。

### 2.2 改前必须执行
1. `graphify query "要改的功能/模块/符号"` → 符号定位（概念模糊时）
2. `codegraph_impact(符号)` → 暴露影响半径，HIGH/CRITICAL 先告知用户等确认
3. `codegraph_context(任务描述)` → 精准拉取上下文，替代读多个文件
4. `codegraph_callers(符号)` → d=1 调用方必须同步检查

### 2.3 改后必须执行
```
npx -y @nodesify/graphify update .
```
确保 graphify 索引与代码一致。

## 三、改代码流程 (Edit Workflow)

### 3.1 铁律
1. **一次只改一件事。** 改完→验证→确认→再改下一件。禁止同时塞多个不相关变更。
2. **简单方案优先。** 先试最直接解法（改一行、加一个参数）。直接方案不行再升级。禁止上来就架构重构。
3. **改前先查。** 涉及框架行为先看文档/源码确认，不凭直觉猜。同一个坑不能踩两次。
4. **改坏了立即回退。** git stash / git checkout 回到上一个正常状态重来。
5. **控制 blast radius。** 改 A 功能不动 B 功能的代码。不要顺手重构、清理、格式化。

### 3.2 反补丁铁律
1. **不重复业务逻辑。** 同一规则已有实现就必须复用。UI/API/store 层发现分散的相同判断，优先抽公共 service/helper/validator。
2. **不破坏分层。** UI=展示交互，API client=请求响应，service=业务流程，domain/validator=可复用规则。
3. **不写死不掩盖。** 禁止硬编码特殊 case、silent catch/return、临时 fallback 掩盖根因。
4. **改前必输出：** 根因 → 调用链 → 现有逻辑分布 → 是否有重复逻辑 → 改哪些文件 → 不动哪些文件 → 风险 → 验证方式。
5. **改后必自检：** 新增重复逻辑？临时补丁？模块不一致？破坏分层？隐藏 fallback？未覆盖边界？需补测试？风险仍存在？

### 3.3 改功能/修 bug 前必须追溯完整数据流
```
输入是什么？(用户手势、点击、事件、数据源)
→ 经过哪些函数/变量逐级转换？
→ 最终渲染/输出的 widget 依赖哪些中间变量？
→ 每一步转换是否正确？有没有隐式的对齐/截断/默认值？
```
禁止凭症状猜根因直接动手。

### 3.4 语义翻译铁律
对本项目代码做任何修改前，必须先出选择题（ABCDE 选项模式）征询用户确认：
- 列出 3-6 个选项覆盖不同方向
- 每个选项说明做了什么、为什么这样做、有什么风险
- 意图明确时至少 3-4 个选项，模糊时 5-6 个
- 禁止用户没回答选择题就动手改代码

**格式：** 不要调 ask_choice 工具。在回复中文字列出 ABCDEF 选项（每行一个，标注改动量和风险），在最下方标注 `猸 推荐组合` 及理由。用户直接打字回复要哪些选项（如 ABD）。

## 五、项目文档维护 (Project Docs)

每次代码/文件/配置/构建/测试类修改后，更新：
- `ARCHITECTURE.md`：技术架构、核心模块、数据流、依赖关系、构建/运行结构、重要实现决策
- `CHANGELOG.md`：修改内容、修改原因、影响文件/模块、后续风险或 TODO

只记录事实，不写入推测性信息。

## 六、调试规则与用户纠正 (Debugging & Correction)

### 6.1 调试铁律
1. **catch 块必须有日志。** 空 catch(_){} = 主动致盲。catch(e){flog('XXX failed: \$e');}。
2. **平台能力必须先连设备看 logcat。** 涉及通知/蓝牙/文件/相机/权限时，只有 logcat 能告诉你真实失败原因。没看 logcat 之前禁止下结论。
3. **改功能/修 bug 前先追溯完整数据流。** 见 3.3。
4. **文件日志机制：** flog() 写文件 + print 双输出，按天分文件，启动时自动清理 >1 天的旧日志。

### 6.2 用户错误纠正
如果用户的陈述、前提、假设或引导方向在事实上是错误的，必须直接、明确地指出，不得：
1. 顺着错误前提继续推理
2. 为了"配合用户"而忽略明显的事实错误
3. 用模糊语言回避矛盾
4. 在错误方向上给出看似合理的建议

## 外部触发词

| 用户说 | 行为 |
|--------|------|
| "跑技能" | 立即 run_skill({name:"force-load-global-memories", arguments:"加载全部全局记忆"}) |
| user 消息中包含 # Skill: force-load-global-memories 或 /force-load-global-memories | 执行完整逐条 recall_memory 流程（22 条） |
| "格式" | 立即停止推理 → 检查 1.1+1.2 → 不满足则删除重写 |
| 指定了 skill 名 | 使用该 skill，禁止自动加载其他 skill |

## 完整流程图

```
用户请求
  ↓
[阶段 1] 回复格式
  ├── 输出 [记忆执行情况] 头
  └── 前缀 [老板]
  ↓
[阶段 2] 如果涉及代码/文件/配置/构建/测试
  ├── 工具可用性检查（门禁）
  ├── graphify query
  ├── codegraph_impact
  ├── codegraph_context
  ├── codegraph_callers
  └── 输出选择题征询用户
  ↓
[阶段 3] 用户确认 → 执行修改
  ├── 追溯数据流
  ├── 反补丁检查
  └── 一次只改一件事
  ↓
[阶段 4] 改后
  ├── graphify update .
  ├── 更新 ARCHITECTURE.md
  └── 更新 CHANGELOG.md
  ↓
[阶段 5] 回复前自检
  ├── [记忆执行情况] 头
  ├── [老板] 前缀
  └── Summary / Files / Tests / Risks / Next
  ↓
输出回复
```
